## 1. OpenCode 项目架构

### 1.1 项目概述

**OpenCode** 是一个开源的 AI 编程助手，专为终端设计，支持多模型提供商（Claude、OpenAI、Google、本地模型等），避免供应商锁定。

- **GitHub**: https://github.com/opencode-ai/opencode
- **官网**: https://opencode.ai/
- **核心特点**: 开源、多模型支持、终端优先、多会话并行

### 1.3 Agent 设计模式

OpenCode 采用**专业化子 Agent**架构：

| Agent 类型 | 职责 | 使用场景 |
|-----------|------|---------|
| `_arch` | 架构规划 | 问题分解、方案设计，不写代码 |
| `_coder` | 代码实现 | 具体编码任务 |
| `_reviewer` | 代码审查 | 代码质量检查、安全审查 |
| `_tester` | 测试验证 | 生成和执行测试用例 |
| `_researcher` | 信息收集 | 文档搜索、代码库分析 |

**子 Agent 调用机制**：
- 主 Agent 可以调用专门的子 Agent 处理特定任务
- 子 Agent 在隔离的上下文中运行，避免污染主会话
- 支持并行启动多个 Agent 处理不同任务

### 1.4 工作流引擎

```yaml
工作流程:
  1. 初始化:
     - 读取项目配置 (.opencode/)
     - 加载模型提供商配置
     - 建立会话上下文
  
  2. 任务接收:
     - 解析用户请求
     - 确定是否需要子 Agent
     - 规划执行步骤
  
  3. 执行循环:
     - 思考 (Thought): 分析当前状态
     - 行动 (Action): 调用工具/执行命令
     - 观察 (Observation): 收集执行结果
     - 反思 (Reflection): 评估是否需要调整
  
  4. 完成:
     - 汇总结果
     - 更新会话记忆
     - 等待用户反馈
```

### 1.5 工具集成方式

**内置工具**：
- `read_file` - 读取文件内容
- `write_file` - 写入/修改文件
- `search_files` - 搜索文件
- `run_command` - 执行 shell 命令
- `git_ops` - Git 操作

**扩展工具**：
- 支持 MCP (Model Context Protocol) 协议
- 可通过插件系统添加自定义工具
- 工具定义使用 JSON Schema 规范

**配置文件示例** (`.opencode/config.json`)：
```json
{
  "providers": {
    "openai": {
      "apiKey": "your-api-key",
      "disabled": false
    },
    "anthropic": {
      "apiKey": "your-api-key",
      "disabled": false
    }
  },
  "tools": {
    "allowed": ["read_file", "write_file", "run_command"],
    "requireApproval": ["run_command"]
  }
}
```

### 2.3 任务分解策略

**Plan-First 模式**：
1. 接收任务后，先生成详细计划
2. 计划保存到磁盘，等待用户确认
3. 用户批准后，按步骤执行

**Orchestrator 模式**：
```
用户："翻译第 5 讲为 Quarto"
│
├─ Plan-first (模式 1): 起草计划，保存到磁盘，获取批准
│   │
│   └─ 用户："批准"
│       │
│       └─ Orchestrator 激活:
│           ├─ Step 1: IMPLEMENT
│           │   └─ 执行计划步骤 (创建 QMD、翻译内容等)
│           │
│           └─ Step 2: VERIFY
│               └─ 运行验证器：渲染 Quarto，检查 HTML 输出
```

### 2.4 工具调用模式

**预定义工具**：
- 文件读写
- 命令执行
- Git 操作
- 网络请求

**MCP 工具集成**：
- 使用 Model Context Protocol 标准化接口
- 支持第三方服务调用
- 需要认证和授权

**工具调用流程**：
```
Agent 决策
    │
    ▼
选择工具 (基于 JSON Schema 定义)
    │
    ▼
生成工具调用参数
    │
    ▼
执行工具 (可能需要用户确认)
    │
    ▼
收集结果
    │
    ▼
更新上下文，继续决策
```

### 2.5 迭代优化机制

**Agent Teams** (实验性功能，v2.1.32+)：
- 需要 Opus 4.6+ 模型
- 启用标志：`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`
- 内置多 Agent 并行协调能力

**迭代循环**：
1. 执行任务
2. 运行验证/测试
3. 如果失败，分析错误原因
4. 调整策略，重新执行
5. 直到成功或达到最大迭代次数

### 2.6 上下文管理

**上下文策略**：
- 隔离会话：每个任务在独立上下文中执行
- 记忆持久化：重要决策和结果保存到磁盘
- 上下文窗口管理：自动摘要和压缩历史对话

**多会话支持**：
- 可并行运行多个 Claude Code 实例
- 每个实例有独立的上下文和状态
- 通过文件系统进行间接通信

## 4. 多 Agent 协作模式

### 4.1 常见协作架构

#### 4.1.1 层级式架构 (Hierarchical)

```
         ┌─────────────┐
         │ Meta-Agent  │  (协调者/大脑)
         │  (Planner)  │
         └──────┬──────┘
                │
       ┌────────┼────────┐
       │        │        │
       ▼        ▼        ▼
  ┌────────┐ ┌────────┐ ┌────────┐
  │Worker 1│ │Worker 2│ │Worker 3│
  │(Coder) │ │(Tester)│ │(Reviewer)│
  └────────┘ └────────┘ └────────┘
```

**特点**：
- 单一协调者负责任务分解和分配
- 工作 Agent 专注执行
- 适合结构化任务

#### 4.1.2 对等式架构 (Peer-to-Peer)

```
  ┌────────┐     ┌────────┐
  │ Agent 1│◄───►│ Agent 2│
  └───┬────┘     └───┬────┘
      │              │
      │    ┌────────┘
      │    │
      ▼    ▼
  ┌────────┐
  │ Agent 3│
  └────────┘
```

**特点**：
- Agent 间直接通信
- 去中心化决策
- 适合探索性任务

#### 4.1.3 流水线架构 (Pipeline)

```
Input → [Agent 1] → [Agent 2] → [Agent 3] → Output
         (规划)     (执行)      (验证)
```

**特点**：
- 顺序处理
- 每阶段有明确职责
- 适合标准化流程

### 4.2 OWL 框架 (Optimized Workforce Learning)

**核心组件**：
- **Domain-Agnostic Planner**: 领域无关的规划器
- **Coordinator**: 协调器，负责任务分配
- **Specialized Workers**: 专业化工作 Agent

**优势**：
- 跨领域泛化能力强
- 支持动态任务分配
- 可学习优化协作策略

### 4.3 协作通信模式

| 模式 | 描述 | 适用场景 |
|-----|------|---------|
| 消息传递 | Agent 间发送结构化消息 | 任务交接、状态同步 |
| 共享黑板 | 共享工作空间/文件 | 协作编辑、信息共享 |
| 发布订阅 | 事件驱动通信 | 异步通知、状态变更 |
| 直接调用 | 一个 Agent 调用另一个 | 专业化服务调用 |

## 5. 任务分解和规划方法

### 5.1 ReAct 模式 (Reason + Act)

```
┌─────────────────────────────────────────────────────────┐
│                    ReAct Loop                            │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  Thought: 分析当前状态，思考下一步                       │
│       │                                                  │
│       ▼                                                  │
│  Action: 选择并执行行动 (调用工具)                        │
│       │                                                  │
│       ▼                                                  │
│  Observation: 收集行动结果                               │
│       │                                                  │
│       ▼                                                  │
│  Reflection: 评估结果，决定继续或调整                     │
│       │                                                  │
│       └─────────────────────────────────┐               │
│                                         │               │
│                                         ▼               │
│                              (如果未完成，回到 Thought)  │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 5.2 Tree of Thoughts (ToT)

**核心思想**：
- 探索多个推理路径
- 评估每条路径的价值
- 选择最优路径继续

**流程**：
```
              初始问题
                 │
        ┌────────┼────────┐
        │        │        │
    思路 A    思路 B    思路 C
        │        │        │
    ┌───┴───┐ ┌───┴───┐ ┌───┴───┐
    │       │ │       │ │       │
  A1  A2  A3 B1  B2  B3 C1  C2  C3
    │       │ │       │ │       │
    ▼       ▼ ▼       ▼ ▼       ▼
  评估    评估 评估    评估 评估    评估
    │       │ │       │ │       │
    └───────┴─┴───────┴─┴───────┘
                │
                ▼
          选择最优路径
```

### 5.3 分解策略模式

| 模式 | 描述 | 示例 |
|-----|------|------|
| 功能分解 | 按功能模块划分 | 前端/后端/数据库 |
| 顺序分解 | 按执行顺序划分 | 步骤 1→步骤 2→步骤 3 |
| 并行分解 | 可并行执行的任务 | 独立模块同时开发 |
| 优先级分解 | 按优先级划分 | 核心功能→次要功能 |

### 5.4 规划最佳实践

1. **明确目标**：确保任务目标清晰可衡量
2. **定义边界**：明确任务范围和约束条件
3. **风险评估**：识别潜在风险和依赖
4. **里程碑设置**：设置检查点进行进度评估
5. **灵活调整**：根据反馈动态调整计划

### 6.2 MCP 协议 (Model Context Protocol)

**核心概念**：
- 标准化 AI 应用与外部工具/数据源的集成协议
- 灵感来自 Language Server Protocol (LSP)
- 支持工具发现、调用、结果返回

**MCP 架构**：
```
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│   AI Model  │◄────►│  MCP Client │◄────►│  MCP Server │
│  (LLM App)  │      │             │      │   (Tools)   │
└─────────────┘      └─────────────┘      └─────────────┘
                            │                    │
                            │                    ▼
                            │            ┌─────────────┐
                            │            │  External   │
                            │            │   Service   │
                            │            └─────────────┘
                            │
                            ▼
                     ┌─────────────┐
                     │ MCP Server  │
                     │  (Files)    │
```

## 7. 迭代反馈循环

### 7.1 基本迭代循环

```
┌─────────────────────────────────────────────────────────┐
│               Iteration Loop                             │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  ┌──────────┐                                           │
│  │  Plan    │                                           │
│  └────┬─────┘                                           │
│       │                                                  │
│       ▼                                                  │
│  ┌──────────┐                                           │
│  │ Execute  │                                           │
│  └────┬─────┘                                           │
│       │                                                  │
│       ▼                                                  │
│  ┌──────────┐                                           │
│  │  Test    │                                           │
│  └────┬─────┘                                           │
│       │                                                  │
│       ├── Success ──► Done                              │
│       │                                                  │
│       ▼                                                  │
│  ┌──────────┐                                           │
│  │ Analyze  │  (分析失败原因)                           │
│  └────┬─────┘                                           │
│       │                                                  │
│       ▼                                                  │
│  ┌──────────┐                                           │
│  │  Adjust  │  (调整策略/修复问题)                       │
│  └────┬─────┘                                           │
│       │                                                  │
│       └───────────────► (回到 Plan 或 Execute)           │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 7.2 自纠正机制 (Self-Correction)

**Agent-R 框架**：
- 训练 Agent 进行自我反思
- 识别自身错误并尝试新策略
- 从失败中学习

**反射 Agent 架构**：
```
┌─────────────────────────────────────────────────────────┐
│              Reflection Agent Architecture               │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  ┌──────────────┐                                       │
│  │   Planner    │  (生成候选计划/行动)                   │
│  └──────┬───────┘                                       │
│         │                                                │
│         ▼                                                │
│  ┌──────────────┐                                       │
│  │   Executor   │  (在真实环境中执行)                    │
│  └──────┬───────┘                                       │
│         │                                                │
│         ▼                                                │
│  ┌──────────────┐                                       │
│  │   Reflector  │  (诊断和纠正错误)                      │
│  └──────┬───────┘                                       │
│         │                                                │
│         └───────────────► (反馈给 Planner)               │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 7.3 自主循环 (Autonomy Loops)

**REFLECTION → EVALUATION → CORRECTION → EXECUTION**

| 阶段 | 活动 | 输出 |
|-----|------|------|
| Reflection | 回顾之前的行动和结果 | 问题识别、经验总结 |
| Evaluation | 评估当前状态和目标差距 | 优先级排序、成功指标 |
| Correction | 制定纠正措施 | 修正计划、新策略 |
| Execution | 执行纠正后的行动 | 行动结果、新观察 |

### 7.4 迭代优化策略

1. **置信度评分**：对每次输出进行置信度评估，低置信度触发重新思考
2. **多轮验证**：关键步骤进行多次验证
3. **渐进式改进**：每次迭代专注于改进特定方面
4. **终止条件**：设置最大迭代次数或满意度阈值

## 9. 错误处理和恢复

### 9.1 错误分类

| 错误类型 | 描述 | 处理策略 |
|---------|------|---------|
| 工具执行错误 | 命令执行失败、API 调用失败 | 重试、降级、替代方案 |
| 理解错误 | 误解用户意图 | 澄清、确认、重新理解 |
| 逻辑错误 | 代码逻辑问题 | 测试发现、自动修复 |
| 资源错误 | 文件不存在、权限不足 | 提示用户、请求授权 |
| 上下文错误 | 超出上下文窗口 | 摘要、压缩、分页 |

### 9.2 恢复机制

**自动恢复流程**：
```
错误发生
    │
    ▼
错误分类和诊断
    │
    ├── 可自动恢复 ──► 执行恢复策略 ──► 继续执行
    │
    └── 需要用户干预 ──► 报告用户 ──► 等待指示
```

**恢复策略**：
1. **重试**：对于临时性错误，自动重试（带退避）
2. **回滚**：撤销失败操作的影响
3. **替代**：使用替代工具或方法
4. **降级**：降低要求，完成部分功能
5. **求助**：向用户请求帮助或确认

### 9.3 文件变更回滚

**OpenCode 的回滚机制**：
- 在执行可能破坏性的操作前创建备份
- 工具执行失败时自动恢复备份
- 支持手动回滚到任意检查点

### 9.4 错误预防和缓解

| 策略 | 说明 |
|-----|------|
| 预验证 | 执行前检查前提条件 |
| 沙箱执行 | 在隔离环境中执行风险操作 |
| 渐进式执行 | 分步执行，每步验证 |
| 超时保护 | 设置操作超时限制 |
| 资源限制 | 限制资源使用量 |

## 总结

本次调研分析了 OpenCode、Claude Code、Cursor、GitHub Copilot Workspace、Devin AI 等主流 AI 编程助手的架构设计和工作流模式。关键发现包括：

1. **多 Agent 架构**是主流趋势，通过专业化分工提高任务质量
2. **Plan-First + 迭代执行**模式平衡了规划和灵活性
3. **标准化工具接口**（如 MCP 协议）降低集成复杂度
4. **自纠正机制**显著提高系统可靠性
5. **透明的用户交互**建立信任，降低使用门槛

对于积木建模项目，建议采用分层 Agent 架构，实现专业化 Agent 团队，建立多层验证循环，并提供透明的决策过程和完善的错误恢复机制。通过渐进式实施，可以快速构建一个可靠、易用、可扩展的 AI 辅助建模系统。
